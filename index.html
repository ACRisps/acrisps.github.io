数据库基础

Posted on 2024-07-29 In 数据库

概念模型

  • 实体（entity）
  • 属性（attribute）
  • 码（key）：唯一标识实体的属性集
  • 实体型：实体名 + 属性名
  • 联系：一对一、多对多、一对多
  • E-R图

关系的完整性

  • 实体完整性

    • 某一属性组的值能唯一标识一个元组，而其子集不能——候选码，选定一个作为主码。候选码里的属性为主属性。

    • 主属性不能取空（NULL）

  • 参照完整性

    • 外码
    • 属性（组）F是外码，与关系S的主码对应，则F上：
      • 或者为空（NULL）
      • 或者为S中的主码值

  • 用户定义的完整性

    • 例如某个属性要取唯一值等

    • 由用户设置

    • not null、unique

    • CHECK

      • CREATE TABLE Student
(...,
 Ssex CHAR(2) CHECK(Ssex IN ('男','女')),
 ...
);


      • /* 删除约束 */
ALTER TABLE Student
    DROP CONSTRAINT c1;

    • 模式名默认为用户名

  • 删除模式

    • DROP SCHEMA test <CASCADE|RESTRICT>

    • UNIQUE、PRRIMARY KEY等

  • 修改基本表

    • ALTER TABLE Student ADD S_entrance DATE; /* 添加入学时间列，类型为DATE */


    • ALTER TABLE Student ADD UNIQUE(Sno); /* 添加约束条件 */

  • 删除基本表

    • 类似删除模式

  • 建立索引

    • CREATE [UNIQUE][CLUSTER] INDEX <索引名> ON <表名>(<列信息>);

  • 删除索引

    • DROP INDEX <索引名>

  • 聚集函数

    • COUNT、SUM、AVG、MAX……：对列操作
    • WHERE子句里不使用聚集函数，适用GROUP BY + HAVING

  • 连接查询

    • SELECT Student.Sno, SC.Cno
FROM Student, SC
WHERE Studnet.Sno = SC.Sno;

  • 嵌套查询

    • /* 查询和小明在一个系的学生 */
SELECT Sno, Sname
FROM Student
WHERE Sdept IN
    (SELECT Sdept
     FROM Student
     WHERE Sname = '小明'
    );

  • 修改数据

    • UPDATE Student
SET Sage = 22
WHERE Sno = '221250075';

  • 触发器只能定义在基本表上，而非视图

  • /* 当对SC的表修改，新成绩增加了超过10%，则记录到另一个表里 */
CREATE TRIGGER TRI_SC_LOG
AFTER UPDATE ON SC
REFERENCING
    NEW ROW AS NEW_ROW
    OLD ROW AS OLD_ORW
FOR EACH ROW
WHEN (NEW_ROW.Grade > OLD_ROW.Grade * 1.1)
    INSERT INTO SC_L
    VALUES(NEW_ROW.Sno, NEW_ROW.Cno, NEW_ROW.Grade, OLD_ROW.Grade);

  • ACID

    • 原子性：不做、全做
    • 一致性：没有做到不做或全做会导致数据库失去一致性
    • 隔离性：事务间不能相互干扰
    • 持续性：永久性，一旦提交，永久更改

  • 故障

    • 事物内部故障
      • 恢复：逆序看log，反向操作该事务所有操作
    • 系统故障
      • 恢复：正向看log，已提交的重做（重复执行log），未完成的撤销
    • 介质故障
      • 恢复：重装-重做
    • 计算机病毒

  • 数据转储

    • 装入后备副本，重新运行运行转储后所有更新事务
    • 静态转储：停数据库，转储
    • 动态转储：转储、用户事务并发 + 日志文件

  • 日志文件

    • 记录事务对数据库更新操作的文件
    • 包括：开始BEGIN TRANSACTION标记、COMMIT|ROLLBACK结束标记、更新操作
    • 作用：
      • 事务故障恢复
      • 系统故障恢复
      • 动态转储必需
      • 由转储状态到故障前状态，不必运行已完成的事务程序
    • 严格按时间顺序；先写日志后写数据库

  • 检查点

    • 记录了：检查点建立时所有活动的事务以及它们日志的地址

    • 重新开始文件记录了检查点记录在日志里的地址

并发

  • 封锁
    • 先加锁，再操作：排他锁-写锁、共享锁-读锁
    • IS、IX、SIX（读，写子）
  • 时间戳、乐观锁、多版本并发控制

NoSQL

  • 阻抗失谐：关系模型和内存中的数据结构之间存在差异
  • SQL有强一致性，不便于大规模分布式集群
  • 分布式的非关系型数据库
  • 仲裁

    • 写入仲裁(write quorum)：

      • 如果发生两个相互冲突的写入操作，那么只有其中一个操作能为超过半数的节点所认可，W>N/2 。即，参与写入操作的节点数(W)，必须超过副本节点数(N)的一半。副本个数又称为“复制因子（对等式分布模型）
      • 只需要向主节点中写入数据（主从式分布模型）

    • 读取仲裁

      • 只有当R+W>N时，才能保证读取操作的“强一致性”。其中，执行读取操作时所需联系的节点数(R)，确认写入操作时所需征询的节点数(W)，以及复制因子(N)（对等式分布模型）
      • n只需从主节点中读取数据（主从式分布模型）
  • 键值数据库
    • 用户配置信息、购物车
  • 文档数据库
    • 事件记录、内容管理系统及博客平台
  • 列族数据库
    • 事件记录、内容管理系统及博客平台、计数器
  • 图数据库
    • 互联数据、社交网络、安排运输路线、分派货物和基于位置的服务、推荐引擎

简单微服务

Posted on 2024-07-29 In 云原生

基于南京大学软件学院云原生实验

目标说明

使用 Spring Boot 开发两个微服务：Admin Service 和 User Service

  1. 两个微服务启动时都注册到 Eureka 服务；
  2. Admin Service 提供一个「增加用户」的 API：对 API 中的用户名和密码要做不为空校验，然后通过服务发现调用 User Service 完成增加用户的操作（接口设计遵循 RESTful 风格）；
  3. User Service 也提供增加用户的 API：完成对数据库中「用户表」的操作；
  4. 为每个项目编写 Dockerfile，并通过 docker build 构建 Admin 服务，User 服务以及 Eurkea 服务镜像；
  5. 编写 yaml，使用 Kubernetes 运行 Admin，User 和 Eureka 服务；
